# コンプライアンス監査リファレンス（日本の個人情報保護法 中心）

このファイルは、クライアントデータ加工の各工程で確認すべき法令・社内規定の観点をまとめたもの。
`mask_tabular.py` の出力監査ログ(audit.json)と突き合わせ、納品前チェックに使う。

> 注意: 本資料は実務上の確認観点の整理であり、法的助言ではない。最終判断は社内法務・有資格者に確認すること。

## 1. 個人情報保護法（APPI）の主要観点

### 個人情報・要配慮個人情報・個人識別符号
- **個人情報**: 氏名、生年月日その他で特定個人を識別できる情報。他情報と容易に照合でき識別できるものを含む。
- **個人識別符号**: マイナンバー、運転免許証番号、パスポート番号など。原則 `drop`。
- **要配慮個人情報**: 人種、信条、病歴、犯罪歴など。取得・取扱いに本人同意が必要。原則 `drop` または厳格な抑制。

### 匿名加工情報 / 仮名加工情報の区別
- **仮名加工情報**: 他の情報と照合しない限り特定個人を識別できないよう加工。内部分析向け。`hash`・`partial` が該当しうる。
- **匿名加工情報**: 特定個人を識別できず、かつ復元不可能に加工。第三者提供のハードルが下がる。`drop`＋`generalize`＋k-匿名性担保が必要。
- 「どちらを作っているのか」をポリシーの `regulation`/`name` と監査ログで明示する。

### 委託先としての義務（自社が他社データを請け負う立場）
- 委託元との契約で定めた利用目的の範囲内でのみ取扱う。目的外利用は不可。
- 安全管理措置（組織的・人的・物理的・技術的）を講じる義務。本スキルは技術的措置の一部。
- 再委託（個人のClaude Pro利用を含む）の可否・条件を委託元契約で必ず確認する。

## 2. 納品前チェックリスト

加工結果と audit.json を見ながら確認する：

- [ ] ポリシーの `regulation` と作成物の種類（仮名/匿名加工）が一致しているか
- [ ] `drop` 指定すべき個人識別符号（マイナンバー等）が残っていないか
- [ ] `unlisted_columns`（ポリシー未記載列）が空、または妥当に処理されているか
- [ ] 自由記述列に `regex_scrub` がかかり、メール・電話等の混入がないか
- [ ] `k_anonymity.pass` が true か。false なら準識別子を一般化したか
- [ ] `hash_salt_used` を案件鍵として安全に保管／案件終了後に破棄する運用が決まっているか
- [ ] 入力原本（生データ）の保管・削除方針が委託元契約に沿っているか
- [ ] 加工後データのみを後続のClaude等に渡す導線になっているか（生データを直接投入しない）

## 3. 過剰隠蔽の回避（データ価値とのトレードオフ）

安全性と分析価値はトレードオフ。全列を `redact` すれば安全だがデータは無価値になる。
- 分析に必要な属性（購入金額、地域、年代など）は `keep` または `generalize` で「粒度を残す」。
- 直接識別子（氏名・連絡先・識別符号）のみ強く落とす。
- 「なぜこの列を残すか」をポリシーの `_notes` か案件メモに記録し、説明責任を果たす。

## 4. GDPR等が絡む場合（海外クライアント時の追加観点）

本スキルは個情法中心だが、海外データが混じる場合は以下を追加確認：
- 処理の法的根拠（同意・正当な利益等）、データ主体の権利（消去権等）。
- 越境移転の制約。個人のClaude Pro経由の処理が移転に当たらないか委託元と確認。
- 詳細は社内法務に必ずエスカレーションする。
